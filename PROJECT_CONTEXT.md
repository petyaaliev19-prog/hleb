# PROJECT CONTEXT

## Что это

`Хлебушек` — сервис предзаказа свежего хлеба и выпечки из локальных пекарен.

Пекарни публикуют ассортимент, доступность по датам и слоты самовывоза.
Покупатели выбирают товары, оформляют заказ и забирают его в пекарне.

Интерфейсы разделены по аудиториям:

1. Покупатель видит публичный каталог пекарен и витрину выбранной пекарни с товарами.
2. Пекарня видит отдельный кабинет для управления карточкой, ассортиментом, слотами и выдачей заказов.

## Цель первой версии

Проверить, готовы ли:

1. Покупатели резервировать хлеб заранее.
2. Пекарни принимать такие заказы через простой веб-интерфейс.
3. Пекарни использовать сервис как инструмент планирования выпечки и выдачи.

## Что входит в текущий продуктовый фокус

1. Каталог пекарен.
2. Страница пекарни с ассортиментом.
3. Выбор даты и слота самовывоза.
4. Предзаказ без регистрации.
5. Оплата только на месте.
6. Отдельный кабинет пекарни.
7. Лента заказов в кабинете пекарни.
8. Панель пекарни со сводкой по заказам.
9. Смена статусов заказа в панели пекарни.

## Что не входит в текущий продуктовый фокус

1. Доставка.
2. Онлайн-оплата.
3. Полноценные рекуррентные подписки.
4. Рейтинги и отзывы.
5. Мобильные приложения.
6. Сложная CRM и логистика.

## Главные бизнес-правила

1. Пользователь видит товары на выбранную дату.
2. Пользователь выбирает слот самовывоза.
3. Оплата происходит только в пекарне.
4. Покупательский интерфейс не показывает операционные ленты заказов и инструменты пекарни.
5. Кабинет пекарни не смешивается с публичным каталогом.
6. Ценность для пекарни — сводка по объему выпечки и статусам выдачи.
7. Ценность для покупателя — гарантированный самовывоз свежего хлеба.

## Текущая реализация

Сейчас основной рабочий слой — `Next.js + SQLite`.

Он уже покрывает:

1. Каталог пекарен.
2. Витрину пекарни.
3. Корзину и оформление заказа.
4. Отдельный owner-раздел пекарни.
5. Ленту заказов в owner-разделе.
6. Панель пекарни.
7. Управление статусами заказов через API и UI.

## Статусы заказа

Текущий поток статусов:

1. `new` -> `confirmed`
2. `confirmed` -> `ready`
3. `ready` -> `picked_up` или `not_picked_up`
4. Отмена доступна до этапа готовности: `new` -> `cancelled`, `confirmed` -> `cancelled`
